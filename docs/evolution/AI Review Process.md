# AI Review Process

Pickla uses multiple perspectives because product truth is not found from one angle.

The goal is not to make AI the decision-maker.

The goal is to make decisions harder to fool.

## The Path

### Idea

An idea begins as a possible improvement.

It may come from a founder instinct, customer behavior, staff pain, usage data, or a failed screen.

### Codex

Codex turns the idea into a concrete artifact:

- An audit
- A plan
- A prototype
- A code change
- A document

Codex should stay close to the existing system and avoid unnecessary invention.

### Architecture Review

Architecture Review asks:

- Does this fit the model?
- Does it preserve one truth?
- Does it create a parallel system?
- Does it weaken future multi-venue or customer identity work?
- Does it add debt that will be expensive to undo?

### Experience Review

Experience Review asks:

- Does the customer understand the moment?
- Does staff know the next action?
- Does the screen reduce cognitive load?
- Does it honor the Constitution?
- Does it feel like Pickla?

### Reality

Reality checks whether the work survives use.

No review is complete until the product meets customers, staff, bookings, check-ins, payments, exceptions, and venue operations.

### Git

Git records the decision.

The work should be understandable later: what changed, why it changed, and what it deliberately did not change.

### Deploy

Deployment sends the decision into the world.

It should be calm, observable, and scoped.

### Weekly Review

Weekly Review converts shipped work into learning.

It asks:

- What did reality confirm?
- What did reality reject?
- What should be deleted?
- What should be promoted to principle?
- What should wait?

## Why Multiple AI Perspectives Help

Architecture sees structure.

Experience sees friction.

Codex sees implementation.

Reality sees truth.

No single perspective is enough.

Better decisions come from productive disagreement.

Consensus is not the goal.

Clarity is.
